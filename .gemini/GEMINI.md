# 多智能体协作全局准则 (针对 Gemini)

作为在此项目中工作的智能体，你**必须**遵守以下协作规则，以确保多智能体同时工作时不会产生冲突。

## 1. 核心协作理念

*   **隔离与互斥**：本项目采用分布式架构，任何修改代码的操作都必须基于对该模块/文件的“独占权”。
*   **沟通先行**：永远假设有其他智能体正在同时修改文件，不确定的情况优先使用工具检查状态或询问用户。

## 2. 必须遵守的硬性规则

1.  **文件修改前必锁**
    *   在你尝试修改、删除或重命名任何业务文件前，必须调用内置的 MCP Server (`RepoManager`) 中的锁定工具（如 `acquire_lock`）锁定该文件或目录。
    *   如果锁定失败，说明有其他智能体正在操作，你必须等待或处理其他任务。
2.  **文件修改后释放**
    *   完成你的任务并在本地验证后，立即提交代码并调用 `release_lock` 释放锁。
3.  **遵循既定的 Skills**
    *   在进行代码同步和提交时，如果项目提供了如 `collaborative_git` 之类的技能（Skill），请严格遵循技能中定义的步骤（Pull -> Rebase -> Test -> Push），严禁使用 `git push --force`。
4.  **规范化输出与日志**
    *   （根据用户全局规则）使用中文输出。
    *   遇到文件长度超过 1000 行，必须主动进行重构与拆分。

## 3. MCP 工具使用规范

*   **RepoManager**: 这是项目自带的 C# 单体 MCP 服务，负责维护整个工作区的“状态板”。你必须时刻与之保持通信，获取项目当前的占用情况。
